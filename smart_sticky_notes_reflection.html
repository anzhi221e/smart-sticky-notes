1. 问题定义

用户原话："我平时经常会需要在各个端进行笔记记录。但是这些笔记往往是散落在各处的。我想要把它们汇总成为一份文件，并让AI读取或者是整理的话会非常困难。"

核心约束

  • 跨端：手机和电脑都能用
  • 语音输入：录音后自动转文字，同时保留原音频
  • 低成本：尽量免费（最终实现为 $0）
  • 本地落盘：最终内容化为 Markdown 文件存到 OneDrive 文件夹
  • 国内可用：用户在中国大陆，访问海外服务有障碍
  • 不关注前端形态：只要能随时随地记录，最终汇总到 Markdown 文件夹

2. 技术调研过程

探索的方案

方案评估结论
微信企业号/测试号 + 后端体验最贴近习惯，但需要服务器、注册审核、消息接口限制放弃
Telegram Bot + 本地拉取技术简单，但国内需代理，违背"随时随地"放弃
PWA + 本地同步服务零成本、国内无障碍，但 PC 需要开机改进
PWA + Supabase + PC 同步脚本加入云中转层，解决 PC 离线问题。免费额度够用采纳

关键技术验证

Web Speech API 限制：设计初稿假设离线录音后可以自动转写，但验证发现 SpeechRecognition 只能处理实时麦克风输入，不能对音频文件做转写。修正：语音功能仅在线可用，离线时禁用并提示使用文字输入。

Supabase refresh_token 一次性使用：尝试让 PWA 和 PC 脚本共享 refresh_token 时发现，每次 refresh 后旧 token 立即失效。结论：PC 脚本使用 service_role key 独立认证，不与 PWA 共享用户态 token。

esm.sh CDN 国内被墙：PWA 使用 esm.sh/@supabase/supabase-js@2 加载 SDK，导致国内用户打开即黑屏。切换到 jsdelivr.net + import map 方案解决。

7. 问题与洞察

CSS !important 的隐秘陷阱

黑屏 Bug 持续了约 30 分钟才定位到。根因：HTML 元素同时有 .screen（display:none）和 .hidden（display:none!important）。JS 切换 .active 时 !important 优先级高于 class 切换，导致所有页面永远不可见。教训：避免在 CSS 中使用 !important，尤其是会被 JS 动态切换的元素。

Supabase Refresh Token 旋转机制

尝试让 PWA 和 PC 脚本共享 refresh_token 时反复遇到 "Already Used" 错误。Supabase 的 refresh_token 是一次性使用：每次 refresh 返回新 token，旧 token 立即失效。这意味着不能把同一个 token 同时给两个客户端用。最终用 service_role 绕过了这个限制，代价是 PC 脚本获得了更高权限。

国内网络环境的现实约束

esm.sh CDN 被墙、Vercel 手机访问慢、Cloudflare 登录被墙——这些都不是代码问题，但决定了用户体验。一个"能用"的系统，在中国大陆需要：可访问的 CDN（jsdelivr）、可访问的云服务（Supabase 恰好可用）、以及 VPN/代理作为兜底。

用户反复确认的关注点：低成本和简洁是最高优先级。用户多次拒绝引入付费服务（Whisper API）、拒绝复杂的初始化流程、选择浏览器内置的免费方案。这不是技术妥协，而是正确的产品决策——MVP 不需要完美，需要能用。

v2 关键洞察

双发送 Bug —— 最顽固的 UI 联动 Bug

根因：发送 → appendChild → scrollTop = scrollHeight → 触发滚动监听 → "接近底部" → loadOlderNotes() → 把刚发的同一条从数据库查回来再次 append → DOM 里两条。

数据库只有一条，UI 渲染了两条。刷新后重新加载只有一条——这就是"清缓存后两条变一条"的原因。

8 次失败尝试：

  • e.preventDefault() + stopPropagation() — 无效，问题不在事件层
  • stopImmediatePropagation() — 无效
  • 切换 keydown → keypress — 无效
  • 时间戳防抖（800ms → 1000ms） — 无效，第二次调用在 12 秒后
  • 重新引入 _isSending 锁 — 无效，锁在第一次完成后已释放
  • setupMainUI 防重复注册 — 无效
  • 禁用 Enter、仅用发送按钮 — 仍然两条
  • 拆分 try/catch + 客户端 UUID 幂等 — 无效，问题在 DOM 层

教训：scrollTop = scrollHeight 是副作用炸弹。任何"滚到底"操作都可能触发已有的滚动监听逻辑。解决方案：去重检查（发送新笔记后跳过一轮滚动加载）、或者滚动监听加 data-note-id 已存在判断。

CSS 优先级——标签不可见之谜

标签 pill 在彩色气泡上不可见，因为 CSS 有两个 .note-bubble .note-tag 规则互相覆盖——后者用 var(--accent-dim) 做背景，在粉色气泡上等于"粉色字放在粉色背景上"。

教训：永远检查合并后的 CSS 规则。两个同选择器规则，后者覆盖前者，不管看起来是否"应该合并"。

Service Worker 缓存——网络优先 vs 缓存优先

开发阶段缓存优先策略导致每次代码更新都需要手动 Unregister SW。改为网络优先（Network First）后，在线时永远拿最新代码。

主题系统复杂度

8 套主题 × 2 种渲染模式（统一色 / 多色）= 16 种视觉效果。CSS 变量 + JS inline style 混用导致切换主题时气泡颜色不一致。最终方案：统一通过 --bubble-bg + --bubble-text CSS 变量控制，多色模式通过 dataset.multi 标记 + inline style 覆盖。
