1. 问题定义

用户原始需求："markdown 语法在页面上的显示是只显示那些比较基础的，对于 callout、quote（>）还有表格（|---|---|）格式语法并不会显示在气泡中，这是为什么？"

核心约束

  • 必须是 零依赖或极轻依赖——PWA 在移动端运行，不能拖慢启动速度
  • 必须兼容 国内 CDN 环境——jsdelivr 可能不稳定，最好本地 bundle
  • 不能破坏现有的标题、加粗、删除线等 已有功能
  • 需要保持 后续自定义语法扩展 的能力（callout 等）
  • 安全：便签内容是用户数据，必须防止 Stored XSS

7. 问题与洞察

Bug 1：HTML 转义破坏 Markdown 语法

现象：blockquote 语法 > text 输入后不显示引用样式。

根因：在 marked.parse() 前做了 HTML 转义，> 变成 &gt;，marked 无法识别为 blockquote 标记。

修复：改为先解析后过滤——用自定义 Renderer 在输出端验证 URL 协议，再用正则剥离危险标签。

教训：Markdown 引擎和 HTML 安全过滤的先后顺序至关重要。转义是输出端的职责，不应在输入端做。

Bug 2：CSS 变量主题冲突

现象：表格表头文字白色不可见（白字 + 白底）。

根因：单色主题（如 blue-light）设置 --bubble-text: #fff（白字），表头用 --surface（白色背景）。--bubble-text 和 --surface 配对不当。

修复：表头显式设置 color: var(--text)——--text 与 --surface 在主题系统中是正确配对的。

教训：项目的 CSS 变量有隐式配对规则。气泡外壳用 --bubble-bg/--bubble-text，气泡内部的 markdown 元素用全局 --text/--surface/--bg。

Bug 3：编辑后不必要全量刷新

现象：编辑笔记保存后屏幕闪烁（"连续频闪"）。

根因：saveEditing() 已用新 HTML 就地更新了气泡（L65），之后又调用 onSaved() → loadNotes() 全量清空重建列表（L558 innerHTML = ''）。

修复：移除多余的 onSaved() 调用。气泡已更新，无需重新加载全部数据。

教训：双写（write-twice）模式容易出现一致性问题和性能抖动。就地更新 + 全量刷新不能同时存在。

Bug 4：工作区下拉菜单溢出屏幕

现象：切换按钮移到左侧后，下拉菜单向左飞出屏幕。

根因：定位代码使用 dropdown.style.right = ...（右对齐），按钮移到左边后 rect.right 很小导致计算值过大。

修复：改为 dropdown.style.left = rect.left（左对齐）。

用户关注点

  • 轻量化焦虑：多次确认 marked.js 是否会让产品变重。用数据说明 10KB gzip + PWA 缓存 = 零感知开销。
  • 自定义语法保留：确认 marked.js extension API 不会与未来自定义 callout 语法冲突。
  • 频闪问题：从"怀疑是 CSS/动画问题"到定位"多余 loadNotes()"的具体根因。

调试教训：CSS 治标 vs 根因修复

本次最重要的技术教训

用户报告了三个 UI 问题：引用块上下空白、蓝色侧边条、气泡内容与元信息之间空白。第一次修改只改了 CSS：

修改类型效果
blockquote border-left 蓝色→白色CSS 样式✅ 颜色正确
blockquote padding/margin 缩减CSS 间距❌ 空白仍在
.note-meta margin-top 6px→2pxCSS 间距❌ 空白仍在
.note-text > :last-child { margin-bottom: 0 }CSS 间距❌ 空白仍在

部署后用户反馈问题一个都没解决。回到系统化调试流程后，用 node -e 验证 marked.parse() 的实际输出：

// marked.parse('> 这是引用') 的实际输出：
"<blockquote>\n<p>这是引用</p>\n</blockquote>\n"

// marked.parse('普通文本') 的实际输出：
"<p>普通文本</p>\n"

真正根因：.note-text { white-space: pre-wrap } 会把 HTML 标签之间的 \n 字符渲染为可见空白行。CSS margin 对此无能为力——空间来自文字内容，不是盒模型。

正确修复：在 renderMarkdown() 中清洗 marked 输出：

// notes.js renderMarkdown() — 在现有安全过滤后追加：
const html = marked.parse(text, { renderer });
return html
    .replace(/<script[\s\S]*?<\/script>/gi, '')
    .replace(/\son\w+\s*=\s*(?:"[^"]*"|'[^']*'|[^\s>]*)/gi, '')
    .replace(/>\s+</g, '><')   // ← 新增：清除标签间空白
    .trim();                                      // ← 新增：清除首尾空白

修复验证：

场景修复前修复后
引用 > 文字<blockquote>\n<p>文字</p>\n</blockquote>\n<blockquote><p>文字</p></blockquote>
普通文本<p>文字</p>\n<p>文字</p>
多行内容<p>行1\n行2</p>\n<p>行1\n行2</p>

标签间空白清除，用户内容中的 \n 保留。一行代码修复两个问题。

教训：修 UI 问题时不要直接从 CSS 入手。从数据流源头追踪：输入 → 处理 → 输出 → 渲染。这次 bug 在"处理→输出"环节（marked 输出含 \n），却连续两次在"渲染"环节（CSS）尝试修复。

专项分析：频闪问题的完整历史

频闪问题在本项目中已出现 3 次，每次根因不同。这已不是偶发 bug，而是一类系统性症状——"页面频繁重绘/布局偏移"——需要在开发流程中建立预防机制。

频闪事件时间线

事件 1：编辑保存后的引用块频闪（本次 session 早期）
触发条件编辑已有笔记 → 添加引用块（> 文字）→ 保存
症状保存后页面开始持续闪烁
根因saveEditing() 中 _editingBubble.innerHTML = html 用原始 markdown HTML 替换了整个气泡，导致 .note-text 包装 div 丢失。CSS 选择器 .note-text blockquote 不再匹配，引用块回退到浏览器默认样式（大边距、默认缩进），气泡尺寸剧烈变化。
修复保存后重建完整气泡结构：.note-text 包装 + .note-meta + audio player + 标签点击事件
教训用 innerHTML 替换 DOM 时，必须重建完整的内部结构，不能只替换内容片段
事件 2：CSS 治标失败（同 session 中期）
触发条件事件 1 的第一次修复尝试（仅改 CSS）
症状引用块上下空白和内容间距仍然存在，修复无效
根因white-space: pre-wrap 导致 marked.parse() 输出的 \n 被渲染为可见空白行。CSS margin/padding 对文字内容产生的空间无能为力。
修复在 renderMarkdown() 中添加 .replace(/>\s+<').trim()
教训UI 间距问题不应从 CSS 入手——从数据流源头追踪：输入 → 处理 → 输出 → 渲染
事件 3：自动刷新导致的整页闪烁（本次）
触发条件用户未进行任何操作，页面在后台打开一段时间后开始频闪
症状整页内容周期性消失后重现（白屏闪烁）
根因loadNotes() 被 setInterval 每 30 秒调用一次，每次执行 list.innerHTML = '' 瞬间清空所有 DOM 节点，然后异步等待 Supabase 查询完成后再重建。从清空到重建之间有明显的白屏间隙。
修复改为增量更新：用 Set 追踪已有笔记 ID，只追加新笔记，不清除现有 DOM。切换工作区时显式传 clearFirst=true。
教训周期性刷新不能用"清空→重建"模式。DOM 应被视为持久化资源，只增不减。

频闪问题系统性分析

三次频闪的共同模式：

// 频闪 = 以下条件之一成立：
// 1. DOM 被周期性清空并重建
list.innerHTML = '';  // ← 事件 3 的根因
